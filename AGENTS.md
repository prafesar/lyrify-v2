# AGENTS.md

## Как Работать С Этим Репозиторием

- Этот проект развивается в основном через промпты для внешнего агента в Google AI Studio.
- Предпочитай небольшие, хорошо ограниченные шаги и изменения с понятной проверкой результата.
- Если задача затрагивает только один-два файла, сначала предложи более простой вариант: пользователь может внести такую правку вручную через UI Google AI Studio.

## Главные Ограничения

- Не меняй серверную часть в этом репозитории и не придумывай backend-реализацию внутри клиента.
- Не считай устаревшую Gemini-логику целевым runtime-path. Основной путь клиента должен идти через внешний API.

## Архитектурные Правила

- AI-граница должна проходить через `src/application/ports/aiPort.ts`.
- Конкретные реализации AI-транспорта должны жить в `src/application/adapters/`.
- Точка выбора активного AI-клиента — `src/application/index.ts`.
- Не размещай transport-specific или provider-specific логику в UI-компонентах, хуках и случайных сервисах.
- Если добавляется новая AI-функция, сначала расширяй `AiPort`, потом адаптер, потом подключай UI.

## Что Считается Правильным Направлением

- Перевод и разбор песни должны идти через внешний API.
- Структурированная подготовка лирики должна идти через внешний preparation route.
- Общий смысл песни должен жить внутри structured lecture / explanation flow, а не в отдельном meaning endpoint.
- Пользовательский UI не должен содержать Gemini-specific терминологию.

## Документация Для Агента

- `docs/worker-api-contract.md` — текущий клиентский контракт внешнего API.
- Если пользователю нужен новый или уточненный server contract, не придумывай его как уже существующий локальный факт: сформулируй клиентские ожидания и явно пометь их как требования к внешнему API.
- Если задача длинная и требует локальных инструкций именно для этого репозитория, можно добавлять новые документы в `docs/`.
- Если документ больше не помогает выполнять текущие задачи агента, его лучше удалить или сократить.

## Текущий Приоритет Реализации

- Текущий приоритет — адаптация клиента к external-API-first lexical preparation flow.
- В этом репозитории клиент только подстраивается под внешний API через `AiPort` и адаптеры.
- На этом этапе не нужно мигрировать FSRS-карточки из `idb-keyval` в SQLite.
- Целевая продуктовая mode-модель:
  - `overview`
  - `vocabulary`
  - `phrases`
  - `style`
- Целевая верхнеуровневая навигация страницы трека:
  - `Lyrics`
  - `Overview`
  - `Words`
  - `Practice`
- `overview` — дефолтная точка входа в песню.
- `overview` должен быть guided entry flow: кратко объяснять смысл трека, помогать понять, на что обратить внимание, и вести пользователя к следующему уместному учебному шагу.
- Не превращай `Overview` в пугающий хаб из множества равноправных вкладок и режимов. Он должен снижать когнитивную нагрузку, а не увеличивать ее.
- Историческая ось `compact | rich` считается legacy-переходом, а не целевой моделью.
- `Words` — отдельная вкладка для всех lexical items трека.
- `Words` — это inventory трека, а не часть breakdown UI.
- В `Words` допускается показывать и отдельные слова, и multi-word lexical items / phrases, если они пришли из canonical lexical layer или были использованы в breakdown как реально учитываемые lexical units.
- `Practice` должен быть полезен даже если пользователь еще ничего не сохранил в FSRS.
- Даже без FSRS `Practice` может предлагать легкие действия вокруг текущего трека: quick review, self-check, mini quiz, revisit unknown items, continue from `Overview`.
- `lecture/fetch` на текущем этапе использовать без cache; lecture payloads могут быть более индивидуальными и чувствительными к изменению prompts.
- Перед новым lecture request клиент должен уметь собирать `existingItems` из уже загруженных mode payloads того же трека и передавать их во внешний API.

## Что Нужно Построить На Текущем Этапе

- mode-aware локальное хранение нескольких вариантов разбора одного трека;
- клиентскую работу с prepared track payload, который приходит с backend;
- локальное хранение lexical items трека и статусов знания пользователя;
- локальные вычисления unknown/known counts по lexical items при открытии трека;
- UX для `Words` и `Practice` поверх новой модели данных.
- guided UX для `Overview`, который направляет пользователя от общего понимания песни к следующему полезному режиму, а не просто показывает еще один длинный разбор.

## Черновая Локальная SQLite-Модель Для Этого Репозитория

На этапе `Phase 1` допустимо вводить эти структуры постепенно, а не все сразу.

### `analysis_variants`

Хранит metadata разных вариантов разбора одного трека.

Минимальные поля:

- `id`
- `track_id`
- `mode`
- `target_language`
- `source_language`
- `status`
- `prompt_version`
- `created_at`
- `updated_at`

### `analysis_variant_payloads`

Хранит JSON payload конкретного варианта разбора.

Минимальные поля:

- `variant_id`
- `payload_json`

### `track_lexical_items`

Локальный список lexical items трека.

Минимальные поля:

- `id`
- `track_id`
- `language`
- `base_form`
- `display_text`
- `kind`
- `normalized_key`
- `created_at`

### `track_lexical_occurrences`

Вхождения lexical items в строках.

Минимальные поля:

- `id`
- `lexical_item_id`
- `line_index`
- `occurrence_index`
- `surface_text`
- `parts_json`
- `spans_json`
- `resolution_status`

### `user_lexical_item_status`

Агрегированное знание пользователя по lexical items.

Минимальные поля:

- `normalized_key`
- `language`
- `status`
- `updated_at`

Допустимые статусы на MVP:

- `new`
- `learning`
- `known`

## Обязательные Архитектурные Правила

- `track_cache` остается UI-snapshot / быстрым кешом и не считается источником истины для аналитики.
- Знание пользователя не должно храниться внутри FSRS-карточек.
- Знание пользователя должно выводиться из отдельного `user_lexical_item_status`.
- `analysis_mode` описывает происхождение анализа или карточки, но не является частью identity знания пользователя.
- Новую mode-aware модель строй вокруг `overview | vocabulary | phrases | style`.
- Если в коде пока существует `compact | rich`, рассматривай это как legacy compatibility layer.
- `compact` концептуально соответствует `overview`.
- `rich` не нужно считать эквивалентом одного из новых учебных режимов.
- Подсветка должна опираться на backend-prepared lexical items и programmatic span resolution.
- Translation и lecture должны отправляться поверх prepared track payload.
- `lecture/cached` не считать активным runtime-path на текущем этапе.
- `existingItems` для lecture — это контекст для уменьшения дублей между `overview | vocabulary | phrases | style`, а не жесткий запрет на повтор.
- Если item повторяется между режимами, новый режим должен по возможности давать новый pedagogical angle.
- Если `Overview` уже показал часть phrases/items, следующие mode payloads не должны механически пересказывать их теми же словами. Клиент должен стараться передавать уже использованные items в `existingItems`.
- Client-side extractor допустим только как fallback или validator, а не как основа продуктовой модели.

## Что Считать Успешным Результатом На Текущем Этапе

- один трек может иметь несколько локально сохраненных вариантов разбора;
- переключение режимов не затирает другие варианты;
- клиент умеет принимать и хранить prepared lexical payload;
- клиент умеет вычислять количество неизвестных lexical items для трека;
- текущий runtime-path через внешний API не ломается.

## Что Не Нужно Делать Сейчас

- не переносить FSRS в SQLite в рамках этой фазы;
- не придумывать backend implementation внутри клиента;
- не ссылаться на недоступные этому агенту внешние репозитории как на локально проверяемый источник истины;
- не привязывать знание пользователя к `mode`;
- не реализовывать идеальные offsets как LLM responsibility;
- не завязывать client flow только на наличие permanent cache на сервере.
